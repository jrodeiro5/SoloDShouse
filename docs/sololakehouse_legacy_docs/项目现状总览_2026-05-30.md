## 2. 当前真实架构

SoloLakehouse 当前是一个 **Docker Compose 单机 Lakehouse 模板**，目标是让
FinLakehouse 这样的产品实体可以从同一套工程基线中拉出。

核心组件仍是：

| 层 | 当前组件 | 作用 |
|---|---|---|
| Object storage | MinIO | S3-compatible storage，承载 Iceberg warehouse、MLflow artifacts、未来 audit bucket |
| Metadata DB | PostgreSQL 17 | Hive Metastore、MLflow、Dagster、Superset 等元数据库 |
| Table catalog | Hive Metastore | Iceberg table catalog |
| Query engine | Trino | 通过 `iceberg` catalog 查询三层 medallion 表 |
| Orchestration | Dagster | `demo_data_flow_job` / `full_pipeline_job` / schedule / sensor / asset check |
| ML tracking | MLflow | `ecb_dax_impact` 实验、参数、指标、artifact |
| Metadata catalog | OpenMetadata | 资产发现与目录入口 |
| BI | Superset | Trino 之上的 SQL/BI UI |
| Operator portal | `make health` | 实体身份、服务健康、demo readiness、核心 UI 链接 |

ADR-020 之后，数据路径变成：

```text
ECB live API + DAX checked-in sample CSV
  -> collectors + Pydantic validation
  -> iceberg.bronze.ecb_rates / iceberg.bronze.dax_daily
  -> iceberg.silver.ecb_rates_cleaned / iceberg.silver.dax_daily_cleaned
  -> iceberg.gold.ecb_dax_features
  -> Trino / Dagster / MLflow / Superset / OpenMetadata
```

重要变化：

- Bronze 不再写 MinIO 裸 Parquet 路径，而是通过 `pyiceberg` append 到 Iceberg 表；
- Silver / Gold 使用 `pyiceberg` overwrite，管控在同一个 HiveCatalog 中；
- `scripts/init-iceberg-namespaces.py` 在 `make up` 中初始化三层 namespace 和 6 张表；
- `make demo` 的验收表是 `iceberg.gold.ecb_dax_features`；
- 旧的 `hive.gold.*` staging CTAS 路径已经不是主数据写入路径。

## 5. FLH 部署判定

### 5.1 可以开始部署的理由

当前满足“开始部署验收”的条件：

1. 主线代码已合并并通过 CI；
2. 本地 `lint/typecheck/test` 通过；
3. all-layer Iceberg 写入路径已经落地；
4. `make up` 会执行 Iceberg namespace/table bootstrap；
5. FLH 专用 `.env` 模板存在；
6. VPS 部署步骤和验收步骤已经写成 runbook；
7. restore drill 模板已经更新到 Iceberg baseline。

因此，下一步可以直接按 `docs/vps-deployment-runbook.md` 从第一步开始部署。

### 5.2 还不能直接宣布正式 live 的原因

正式 live 前还差实体级证据：

| 门槛 | 为什么重要 |
|---|---|
| `ENV_FILE=/opt/finlakehouse/.env make verify` 全部 PASS | 证明 FLH 这台 VPS 的服务可用，不只是模板可用 |
| `ENV_FILE=/opt/finlakehouse/.env make pipeline` 成功 | 证明完整 Dagster + MLflow 路径可用 |
| `ENV_FILE=/opt/finlakehouse/.env make demo` 成功 | 证明核心 Bronze/Silver/Gold/Trino 验收路径可用 |
| Trino 查到 Bronze/Silver/Gold 三层数据 | 证明 Hive Metastore + Iceberg + MinIO 状态一致 |
| MLflow 有 `ecb_dax_impact` run | 证明 artifact bucket 和 tracking DB 可用 |
| Iceberg restore drill PASS | 证明灾难恢复路径覆盖 Hive Metastore + MinIO warehouse + Postgres dumps |

这不是代码风险，而是运营责任边界：模板已经可以部署，实体还需要跑自己的验收。

## 6. 推荐部署顺序

按 `docs/vps-deployment-runbook.md` 执行：

1. 准备 VPS：Ubuntu 22.04 LTS 或 Debian 12，至少 4 vCPU / 8 GB RAM / 40 GB SSD；
2. 创建 `/opt/finlakehouse/{app,data,backup,logs}`；
3. clone 仓库到 `/opt/finlakehouse/app`；
4. 确认部署坐标是 `main@0f38e08` 或之后的 main 提交；
5. 从 `docs/finlakehouse-env-template.env` 生成 `/opt/finlakehouse/.env`；
6. 替换所有 `CHANGE_ME`；
7. 将 `/opt/finlakehouse/app/docker/data` 指向 `/opt/finlakehouse/data`；
8. 执行 `ENV_FILE=/opt/finlakehouse/.env make up`；
9. 执行 `make verify`、`make pipeline`、`make demo`；
10. 手工验收 Trino / Dagster / MLflow / MinIO / OpenMetadata / Superset UI；
11. 执行并记录 Iceberg restore drill；
12. restore drill PASS 后，再把 FLH 标记为可持续运行实体。

## 7. 当前仍然诚实保留的边界

这些不是部署阻塞，但部署后要知道它们存在：

| 边界 | 当前状态 |
|---|---|
| Object store | 仍使用 MinIO；SeaweedFS / Ceph / 云 S3 迁移没有纳入本次部署 |
| Secrets | 仍使用 `/opt/finlakehouse/.env`；未接 Vault / KMS |
| Runtime | 仍是单机 Docker Compose；未上 K8s / Helm / Terraform |
| Alerting | 没有主动告警；依赖 `make verify`、portal、人工巡检 |
| Audit evidence pack | v2.6 计划；当前 audit bucket 已配置，但 pipeline 还不写证据包 |
| OpenMetadata restore | 当前策略是服务恢复 + re-ingest，不保证历史 UI 操作记录完整迁移 |
| DAX 数据 | 仍是仓库内 sample CSV，不是实时 DAX API |
| Release tag | 最新部署基线在 `main@0f38e08`；如果需要严格 release artifact，建议再打一个 post-ADR-020 tag |
