## 1. 项目定位（What it actually is — 2026-05-19 版本）

SoloLakehouse 是一个 **可运行、可阅读、云中立、单机本地优先** 的 Lakehouse 参考实现，
现在它的角色已经从"reference impl"演化为 **"上游产品模板（upstream product template）"**。

它**不是**：
- 一个框架或库；
- 一个面向终端用户的 SaaS；
- Databricks / Snowflake 的"开源克隆"；
- 一个生产级、多环境、HA、托管的 Lakehouse 服务；
- 一个永远以 `sololakehouse` 这个名字 24/7 运行的产品实例。

它**是**：
- 一份在 **单台 Docker Compose 主机** 上完整跑通的 Lakehouse 架构样板；
- 用纯开源组件（MinIO / Postgres / Hive Metastore / Trino / Iceberg / Dagster / MLflow / OpenMetadata / Superset）演示现代 Lakehouse 内部机制；
- 一个**可被 fork 出多个产品实体的"模板仓库"**，例如 `finlakehouse`、`aviation-lakehouse`；
- 围绕金融市场数据（ECB 利率 + DAX 指数）演示 Bronze → Silver → Gold → ML 的端到端链路；
- 一个以 **平台工程判断力** 为陈述对象的作品（可被审阅、运行、批判、扩展）。

仓库层级关系（拉 flh 之前的目标心智模型）：

```text
SoloLakehouse  ← 上游模板（this repo）
   ├── 产品实体: finlakehouse        ← 即将拉出（Phase 2，下一步）
   └── 产品实体: aviation-lakehouse  ← 待 finlakehouse 稳定后再拉
```

**关键认知差异**（与 2026-05-05 / 05-09 快照相比）：

| 维度 | 2026-05-05 | 2026-05-19（今天） |
|---|---|---|
| `SoloLakehouse` 定位 | "reference implementation" | "upstream product template" |
| `sololakehouse` bucket / project 名 | 默认产品身份 | 仅是模板默认值，可被 `PRODUCT_ID` 覆盖 |
| MinIO 角色 | 默认对象存储 | "当前使用的 S3 兼容 provider"，与产品身份解耦 |
| 实体拆分 | 概念性讨论 | **Phase 1 模板就绪已闭环（见 §10）**，Phase 2 可开工 |
| `make demo` / 演示路径 | 不存在 | **存在并且是 v2.5 验收门槛之一** |
| 操作 portal | 不存在 | `make health` 启动的轻量 portal 已存在 |
| 备份/恢复演练 | 不存在 | 已落地 runbook + 一次成功 drill 证据 |

## 4. 数据流（Medallion 全链路）

```
ECB SDW REST API        DAX sample CSV (data/sample/)
        │                       │
        ▼                       ▼
Python collectors（ingestion/collectors/）
   - Pydantic v2 schema 逐条校验
   - 不合规记录写 rejected 路径，绝不静默丢弃
        │
        ▼
Bronze Parquet（按 ingestion_date 分区，append-only，永不就地更新）
        │  (bronze/ecb_rates/... / bronze/dax_daily/...)
        ▼
Silver Parquet
   - 类型清洗、去重、排序
   - 派生字段：rate_change_bps（ECB）、daily_return（DAX）
        │
        ▼
Gold Parquet（ECB + DAX 时间对齐 join，事件研究特征）
        │
        ├── Hive 外表：hive.gold.ecb_dax_features          (Trino REST 注册)
        └── Iceberg Gold：iceberg.gold.ecb_dax_features_iceberg (Trino CTAS)
                  │
                  ├── Superset 仪表盘 / ad-hoc SQL
                  ├── OpenMetadata 资产 + 血缘 UI
                  └── MLflow（XGBoost / LightGBM + TimeSeriesSplit + MinIO artifacts）
```

**桶布局**：

| 桶 | 用途 | 当前默认 | 实体级覆盖 |
|---|---|---|---|
| `DATA_BUCKET` | Bronze / Silver / Gold Parquet | `sololakehouse` | flh: `finlakehouse-data` |
| `AUDIT_BUCKET` | v2.6 lineage / WORM 证据（占位） | `sololakehouse-audit` | flh: `finlakehouse-audit` |
| `MLFLOW_ARTIFACT_BUCKET` | MLflow 模型 artifacts | `mlflow-artifacts` | flh: `finlakehouse-mlflow` |
| `WAREHOUSE_URI` | Hive/Trino warehouse 根 URI | `s3a://sololakehouse/warehouse/` | flh: `s3a://finlakehouse-data/warehouse/` |

**当前真实数据边界**：

- ECB 是真实 live REST API；
- DAX 是 **仓库内固化的 sample CSV**，不是实时市场 API；
- v2.5 已具备 Iceberg Gold，但 **v2.6 要求的"三源 lineage evidence pack"尚未实现**；
- audit bucket 已 **配置存在**但 **尚未被 pipeline 写入** 任何证据文件。

代码位于 `dagster/`。

```
    │                │
ecb_silver       dax_silver
       \            /
```

**已交付形态**（`dagster/definitions.py`）：

| 类型 | 名字 | 备注 |
|---|---|---|
| Job | `full_pipeline_job` | v2.5 的主 pipeline（含 ML） |
| Job | `demo_data_flow_job` | acceptance 演示用，剔除 ML，仅跑数据流 |
| Schedule | `daily_pipeline_schedule` | `0 6 * * 1-5` UTC，工作日 06:00 |
| Sensor | `ecb_data_freshness_sensor` | 每 30 分钟一次，lag ≥ 48h 触发 `ecb_bronze` |
| Asset Check | `gold_features_min_rows_check` | Gold 行数 ≥ 10 才通过 |
| Resource | `MinioResource` / `PipelineConfigResource` | 全部从 env / `storage_config.py` 读 |
| IO Manager | `ParquetIOManager` | 可选，给将来 DataFrame-native asset 备用 |

**运行入口**：
- `make pipeline` 走 webserver 容器执行 `full_pipeline_job`；
- `make demo` 走同样的容器执行 `demo_data_flow_job`；
- Dagster UI：`http://localhost:3000`，由 `dagster-webserver` 服务提供；
- 持久化：`dagster_storage` PG 数据库 + `docker/data/dagster/` bind mount。

**Phase 2 视角**：flh 拉出后，Dagster 实例 **应当 entity-local**——不要让 finlakehouse 与未来的 aviation-lakehouse 共用 Dagster 持久化目录或同名 schedule。`task.md` Phase 2 已明文要求"Dagster instance: entity-local"。

## 7. 轻量 SLH Portal（`make health`）

- 实现：`scripts/health-server.py`（`ThreadingHTTPServer`，纯标准库 + `runtime_identity` + `storage_config` + `verify-setup.py` 输出）；
- 入口：`make health`（默认端口 `8090`），`make health-json` 走 JSON 模式；
- 访问：`http://127.0.0.1:8090/health`。

它回答的两个问题：

```
1. 当前 lakehouse 实体是不是健康的？
```

## 13. 设计哲学（隐式规则，写出来让 flh / aviation 不要破坏）

1. **可读性是一等公民** —— 全栈跑在单机上是个硬约束，不为"看起来更专业"引入额外组件。
2. **失败必须可见** —— Pydantic 校验失败、quality check 失败、asset check 失败一律抛异常，禁止 silent degrade。
3. **元数据外置** —— Hive Metastore 是 "什么表 / 在哪 / 什么 schema" 的唯一真相源，元数据不藏在脚本里。
4. **Bronze 不可变** —— 永远 append 新分区；修正走 silver/gold 派生，不就地改 bronze。
5. **凭据走环境变量 + envsubst** —— 仓库里不留明文密钥，`config/trino/catalog/*.properties` 是模板。
6. **时序就是时序** —— `TimeSeriesSplit`，禁止 random CV、禁止 future-leak。
7. **每个 ADR 写下被拒选项** —— "选了 X" 必须配 "拒了 Y 因为 Z"。
8. **实体身份与基础设施身份解耦**（2026-05 新增） —— `PRODUCT_ID` 是产品身份，MinIO 是当前对象存储 provider，二者不绑定。
9. **拆分先于替换**（ADR-019） —— 在拆分实体的同时替换底层组件 = 故障难以归因。

## 14. 真实边界（今天还没就绪的事）

为避免 flh 拉出后立刻撞 SoloLakehouse 自身的坑，下面这些缺口要明确记下来：

### 14.1 v2.5 模板层面

| 缺口 | 后果 / 影响 flh 的方式 |
|---|---|
| 多环境（dev / staging / prod）模型不存在 | flh 上 VPS 后只有"local"形态；上 prod 的方式当前是"手工 fork 配置 + 跑 setup" |
| Promotion / rollback 流程仍是叙事 | flh 的 release 流程目前等同 "git tag + 重启栈"，没有可执行的 promotion checklist |
| 密钥管理仍走 `.env` | flh 上 VPS 后 `/opt/finlakehouse/.env` 即是凭据来源，未接 Vault / KMS |
| SLO / alerting 不存在 | flh 跑挂了不会主动报警，只能靠 portal + verify 巡检 |
| 数据集 governance 契约（owner / SLA / quality_class / consumers）未在仓库内固化 | flh 的 fin.* dataset 进 OpenMetadata 时缺一个对应的 contract 文件 |
| ML 五元组 lineage（snapshot_id × dagster.run_id × feature_version × code_commit × data_contract_hash）未实现 | flh 的 MLflow run 可以追 params/metrics，但还不能直接回答 "用了哪一个 Iceberg snapshot" |
| K8s / Helm / Terraform 路径全部停留在 ADR | flh / aviation 上 VPS 后短期内只能是 Compose；多实体编排自管 |
| Audit 桶已建但未写入证据 | v2.6 lineage evidence pack 还是计划，不是事实 |
| `make verify` 失败原因之一仍是端口冲突（2026-05-09 快照已警示） | flh 上 VPS 时如果该 VPS 上跑了 Airflow / 旧 Superset，要先确认端口隔离 |

### 14.2 实体拆分层面

| 缺口 | 备注 |
|---|---|
| 真正的实体专属 release 流程未定义 | flh 初期建议沿用模板的 `vX.Y.Z` tag + 自己的 `flh-vX.Y.Z` 运行版本号 |
| FinLakehouse / Aviation 各自的 domain contract（aviation 尤甚） | aviation 不能简单复用 fin 的 dataset 命名，需独立设计；fin 短期可以等价于现状 |
| 长期运行的 freshness / cost / 容量观测 | 现在所有"long-running"特性都未在模板上验证过 |

## 15. 拉出 flh 的可执行下一步（不是 roadmap，是清单）

按 `task.md` Phase 2 §1～§2 顺序：

1. 准备 VPS，建立目录布局：
   ```
   /opt/finlakehouse/{app,data,backup,logs}
   /opt/finlakehouse/.env
   ```
2. `git clone` 模板仓库到 `/opt/finlakehouse/app/`（或解包 release artifact）。
3. 从 `.env.example` fork `.env`，覆盖至少：
   ```
   PRODUCT_ID=finlakehouse
   PRODUCT_DISPLAY_NAME=FinLakehouse
   COMPOSE_PROJECT_NAME=finlakehouse
   RUNTIME_VERSION=flh-v0.1.0   # 自定
   TRINO_USER=finlakehouse
   DATA_BUCKET=finlakehouse-data
   AUDIT_BUCKET=finlakehouse-audit
   MLFLOW_ARTIFACT_BUCKET=finlakehouse-mlflow
   WAREHOUSE_URI=s3a://finlakehouse-data/warehouse/
   # 同步覆盖凭据 / S3_ENDPOINT / TRINO_URL / MLFLOW_TRACKING_URI / Superset secret
   ```
4. 执行 `make setup`（一键冷启动）。
5. 跑 `make verify` + 打开 `http://<vps>:8090/health` 校验实体身份显示为 `finlakehouse`。
6. 跑 `make demo`，确认 Hive Gold 与 Iceberg Gold 在 `finlakehouse-data` 桶下生成。
7. 至少跑一次 `docs/entity-backup-restore-runbook.md` 中的 backup → disposable restore 演练。
8. 跑一次 `make pipeline`，确认 MLflow artifact 落到 `finlakehouse-mlflow`。
9. 把 OpenMetadata 的 Trino service 重命名为 `finlakehouse-trino`，Superset 数据库 label 走 `finlakehouse_*`。
10. 至少完成一次完整 schedule 周期（工作日 06:00 UTC），并把行数趋势记入 portal 之外的 release 评估（手工或写入 issue）。
11. 仅在 finlakehouse 完成一次完整 schedule 周期之后，才开始考虑 aviation-lakehouse。

**禁忌（在 flh 上 VPS 之前/之间不要做）**：

- 不要顺手把 MinIO 换成 SeaweedFS / Ceph（ADR-019 明确禁止把这件事和实体拆分耦合）；
- 不要顺手开 v2.6 lineage evidence 实现（v2.5 frozen baseline 还是评估依据）；
- 不要让 flh 与未来 aviation 共享 Dagster / OpenMetadata / Superset 实例；
- 不要修改模板仓库里和实体无关的代码 —— flh-specific 改动走 flh 自己的 layer，模板只接受 generic 改进。
