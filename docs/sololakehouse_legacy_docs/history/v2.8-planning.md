## 1. 目标与约束

### 目标

- 任何 MLflow run 都能反查到："吃了哪份 Iceberg 数据 + 哪条 Dagster 编排 + 哪个代码 commit + 哪份数据契约"
- 训练结束自动生成符合 **EU AI Act Art.13** 的 model card（markdown + PDF）
- ML asset check：模型性能跌出阈值 = pipeline 红
- 把"训练 / 验证 / 测试切分纪律"从代码里的 `TimeSeriesSplit` 调用，提升为 `docs/ml-discipline.md` 的成文规则

### 不做的事（Non-goals）

| 不做 | 留给 |
|---|---|
| 模型 serving（Triton / vLLM / sagemaker-style endpoint） | 永远 out（ADR-011） |
| 推理时的 `decision_evidence_id` | v3.1（E20） |
| ComplianceRAG / AuditCopilot / LineageNarrator | v3.1+ |
| 子群指标（fairness across protected groups）的自动计算 | 数据集本身没有合适字段；保留接口但不实现 |
| 在线特征仓 / Feature store | 永远 out（演示规模无意义） |
| AutoML / 超参搜索基础设施 | v3.0+ |
| Kubernetes 上的 ML 训练 | v3.0+ |

### 约束

- 不引入新平台服务（沿用 v2.5 已有的 MLflow + PostgreSQL + MinIO）
- 训练时长不允许显著劣化（model card 生成不能超过训练时间的 5%）
- Model card 必须**离线可查**（不依赖 MLflow UI 服务在线）

## 2. 当前痛点

- MLflow run 与 Gold 数据版本是"间接关联"（靠时间戳），没有强绑定
- 如果有人问"这个模型用的训练数据是 2026-04-15 的快照对吗"——目前要爬 logs 拼凑
- 训练代码里写了 `TimeSeriesSplit`，但**为什么不能用随机 CV** 这件事没有成文，是部落知识
- EU AI Act 高风险系统义务在 2026-08-02 适用，模型卡概念在仓库里完全不存在
- 模型性能没有 asset check 兜底，劣化不会让 pipeline 变红

## 3. 方案

### 设计：ML 全链路 lineage 五元组

每次 `mlflow.start_run()` 自动 log：

```python
{
  "iceberg.snapshot_id": "<gold table snapshot at training start>",
  "dagster.run_id":     "<asset run id, when invoked from Dagster>",
  "feature_version":    "<dataset_id from data contract>",
  "code_commit":        "<git rev-parse HEAD>",
  "data_contract_hash": "<sha256 of governance/datasets/X.yaml>"
}
```

这五项任何一项缺失 = 训练 fail-fast，不允许"匿名"模型 run。

### 设计：Model card 自动生成

`ml/generate_model_card.py` 在训练结束时被 hook 调用，输出：

```
audit/models/<model_id>/<run_id>/
  card.md                ← markdown 主体
  card.pdf               ← 用 weasyprint 渲染
  signed_manifest.json   ← PGP-signable
  attachments/
    confusion_matrix.png
    feature_importance.png
    cv_fold_metrics.csv
```

字段对齐 EU AI Act Art.13 与内部模型治理证据口径：训练数据来源 / 切分方式 / 性能指标 / 已知偏差 / 用途与禁用场景 / 监控建议。

### 设计：EU AI Act Art.10 evidence form 边界

v2.8 提供 Art.10 所需的 evidence form：训练数据来源、训练 / 验证 / 测试切分、质量检查、代表性说明、已知限制与记录保存位置。它只保证字段结构、产物路径和审计可追溯性，**不**自动验证客户填写内容是否充分、准确或构成合规；内容验证仍由客户的数据治理、模型风险管理和法务流程负责。

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| Iceberg snapshot_id 在并发训练时可能漂移 | lineage 失真 | 训练入口先 `iceberg.snapshot()` pin 一个固定 snapshot，整个 run 锁定该 snapshot |
| Model card 字段越加越多，训练后 hook 时间膨胀 | 训练性能劣化 | 报告生成异步落盘；只有 PGP 签名同步阻塞（< 1 秒） |
| weasyprint 在某些 Linux 镜像下字体缺失 | PDF 渲染失败 | 在 docker/dagster 镜像中预装 fonts-noto；PDF 失败时自动回退到 markdown-only |
| EU AI Act 字段解读争议 | 模型卡反复修改 | 字段对照表注明"基于公开法规文本与客户适用监管场景的项目内解读"，不对外承诺合规背书 |
| ML asset check 阈值过严 → pipeline 频繁红 | 团队信任下降 | 阈值采用"过去 7 次成功 run 的均值 - 2σ" 自适应，避免硬编码 |

## 8. 技术债与延后项

- 接受：子群指标暂不计算（数据集没合适字段）
- 接受：模型卡只覆盖训练侧，推理侧 evidence id 留 v3.1
