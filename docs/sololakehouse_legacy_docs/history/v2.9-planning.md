# v2.9 规划笔记 — 运维就绪（Operational Readiness）

## 版本

- 目标版本：v2.9
- 规划窗口：2026-Q4 — 2027-Q1 上半段
- 状态：草案
- 前置：v2.6 / v2.7 / v2.8 三轮证据流均已就位
- 配套文档：[`企业级演进规划_2026-05-05.md`](../企业级演进规划_2026-05-05.md)

## 在演进路径中的位置

```
v2.5 (已交付)         "平台能跑"
   ▼
v2.6                 "平台能产出证据"        P1: 监管血缘
   ▼
v2.7                 "平台能证明开放性"      P2 + P4: 主权与多引擎
   ▼
v2.8                 "平台能产出可审计 AI"   P5 前置: ML 合规绑定
   ▼

   ▼
v3.0 (目标)          "平台跑在生产上"        K8s + Helm + Terraform
```

**主线**：在 Compose 上把 v3.0 K8s 所需的"生产化形状"先立起来——SLO、密钥纪律、晋升与回滚证据、就绪检查——让 v3.0 是**纯粹的运行时迁移**，而不是"迁移 + 新建治理叙事"的混合工作。

> 一句话原则：**Compose 不变，纪律先到位。**

## 1. 目标与约束

### 目标

- 落地最小 SLO 集，每次 pipeline 跑完自动更新到 PostgreSQL `slo_metrics` 表
- Superset 上有"平台健康"仪表盘，监管/团队成员可一目了然
- 本地密钥纪律明确：`.env.shared` vs `.env.secrets`，secrets 不出现在任何 git 历史里
- 晋升与回滚 checklist 有 `make` 入口，即使现在只有一个环境，证据形态先固化
- Iceberg snapshot 层做一次回滚演练并归档
- `scripts/check-k8s-readiness.py` 检查所有服务是否容器化就绪（健康端点、配置外置、状态外置）—— v3.0 启动前必须全绿

### 不做的事（Non-goals）

| 不做 | 留给 |
|---|---|
| 实际部署到 Kubernetes | v3.0 |
| Helm chart | v3.0 |
| Terraform module | v3.0 |
| 多环境（dev/staging/prod 真实分离） | v3.0 |
| Vault / cloud secret manager 集成 | v3.0 |
| Prometheus / Grafana 接入（SLO 数据先落 PG，告警走 Dagster sensor） | v3.0+ |
| ArgoCD / FluxCD GitOps | v3.0+ |
| 自动 incident response | v3.0+ |
| 推理证据 ID / ComplianceRAG | v3.1+ |
| 模型 serving | 永远 out（ADR-011） |

### 约束

- Compose 不变；引入新组件的 PR 默认拒绝
- SLO 计算开销不能进入 hot path（pipeline 跑完后异步计算）
- 密钥纪律的执行通过 pre-commit hook，不依赖人记得

## 2. 当前痛点

- v2.6 / v2.7 / v2.8 都产出了证据，但 **"证据本身的健康度" 没有人盯**——审计写入失败、model card 没生成、lineage CLI 超时，都需要人为发现
- 仓库的 `.env` 把所有变量混在一起，新人 onboarding 时容易把 secrets commit 进去
- "回滚"在 ADR 里有叙事，没有任何 `make` 入口
- v3.0 启动前没有 readiness 自查工具，可能开始 K8s 后才发现某个服务没有 health endpoint

## 3. 方案

### 三条工作流

```
1. SLO emit + dashboard          (Block C 落地)
   slo_metrics PG 表
   Dagster asset hook 写入
   Superset dashboard "Platform Health"
   Dagster sensor 触发"SLO breach" 事件

2. Secrets discipline            (Block D 第一阶段)
   .env.shared (commit)
   .env.secrets (gitignore + pre-commit hook)
   masked logging convention
   secrets rotation drill (本地模拟)

3. Promotion / rollback shape    (Block B 落地形态)
   make promote-evidence            (单环境也产出证据包)
   make rollback-iceberg DATASET=X SNAPSHOT_ID=Y
   scripts/check-k8s-readiness.py
```

### 关键决策

- **SLO 数据先落 PG，不引 Prometheus**：Prometheus 接入是 v3.0 的事；v2.9 的目标是"指标存在且可查"，不是"指标走标准协议"
- **告警靠 Dagster sensor**：每 5 分钟扫一次 `slo_metrics`，breach 时 log + 写 `audit/incidents/`；不接 PagerDuty / Slack
- **晋升 checklist 单环境也跑**：哪怕 from=local to=local，也走完一遍流程，把产物形态固化
- **Readiness check 是 v3.0 的入场券**：不绿，v3.0 不能开工

### 与 ADR-015 的交叉验证

ADR-015 已明确 v3 观测栈采用 Prometheus + Grafana + Alertmanager。v2.9 仍选择先把 SLO 落到 PostgreSQL，是因为本版约束是 Compose 默认栈不变、拒绝新增常驻平台服务；目标是固化指标语义、计算窗口、阈值和证据包引用，而不是提前引入 K8s 时代的采集链路。`slo_metrics` 表必须按后续 Prometheus exporter / Pushgateway 可映射的字段设计：metric name、labels、window、value、threshold、status、source_run_id、evidence_uri。这样 v3.0 接入 Prometheus 时迁移的是传输与存储后端，不重写 SLO 定义。

## 6. 给下一个版本（v3.0）留下了什么

- **运行时迁移即可**：v3.0 的工作量从"K8s + 治理 + 多环境 + 密钥"压缩为"K8s + 多环境 + Vault"——治理叙事和 SLO 已就位
- **Helm values 模板**：v2.9 已把所有配置外置，每个服务的 env / volume / port 已经是"K8s-shape"，Helm 直接复用
- **Promotion 与 rollback 的产物形态**：v3.0 多环境晋升只需把 `make promote-evidence` 接到 GitHub Actions，证据 schema 已固化
- **Readiness 自查脚本**：v3.0 的每一次 chart 升级都先跑 readiness check，是入场券

## 7. 风险与缓解

| 风险 | 影响 | 缓解 |
|---|---|---|
| SLO 阈值定得过严，每次 pipeline 都触发 breach | 团队对告警麻木 | 阈值基于过去 30 天分位数自适应；初版宽松，逐步收紧 |
| `.env.secrets` 拆分破坏现有 `make up` | onboarding 体验下降 | Makefile 自动 merge 两个文件；提供 `make init-secrets` 引导生成 |
| Iceberg 回滚演示在生产数据上失误 | 真实数据丢失 | 仅在 demo 数据集上演练；rollback CLI 加 `--confirm` 强制确认 |
| Readiness check 拦下太多 v3.0 工作 | v3.0 推迟 | 拦下 = 在 v2.9 内修复，不允许带病进入 v3.0；这正是 v2.9 存在的意义 |
| 密钥轮换演练在本地无法模拟真实云密钥服务 | 演练价值打折 | 演练范围限定于"轮换流程是否可执行"，不假装是云端 |

## 8. 技术债与延后项

- 接受：本版没有真实多环境，只有"单环境跑流程"
- 接受：告警不接 PagerDuty / Slack，仅写 `audit/incidents/`
- 延后到 v3.0：把 SLO 指标走 Prometheus 标准协议
- 延后到 v3.0：Vault 集成与云密钥管理
- 延后到 v3.0：CI/CD 触发的自动晋升（v2.9 的 make 入口仅本地手动跑）
- 延后到 v3.0+：incident response 自动化（runbook → action）

## 附：四个小版本的演进总览

| 版本 | 主题 | 一句话能力 | 关键产出 |
|---|---|---|---|
| **v2.6** | 治理证据底座 | "出血缘证据 < 60 秒" | 数据契约 + lineage CLI + WORM 审计 |
| **v2.7** | 主权与开放性证据 | "4 引擎读同一张表，5 分钟内" | 多引擎 demo + 主权报告 + 退出剧本 |
| **v2.8** | ML 合规底座 | "每个模型 run 都有 EU AI Act 模型卡" | MLflow lineage 五元组 + 自动模型卡 |
| **v2.9** | 运维就绪 | "SLO 看板亮、回滚可演练、K8s 入场券绿" | SLO + 密钥纪律 + 晋升/回滚 + readiness |

四个小版本的共同基因：
- **都不离开 Compose**
- **都不引入新平台服务**（最多引入演示 profile）
- **都把"宣称的能力"变成"可执行命令 + 可签名产物"**
- **都为 v3.0 留下可复用接口**

读 v3.0 的人应该这样理解 v2.6–v2.9：**它们不是 v3.0 的"前菜"，它们是 v3.0 的"承重墙"。** v3.0 之所以可能在合理时间内交付，是因为治理、开放性、ML 合规、运维纪律已经在 Compose 上跑通；v3.0 的工作就**只是把这些能力换一个运行时**。
