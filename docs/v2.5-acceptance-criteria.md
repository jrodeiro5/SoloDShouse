## Purpose

This document defines the **Definition of Done** for SoloLakehouse v2.5.

It is not a feature checklist. It is a **trust-establishment checklist**: the bar a third party (interviewer, contributor, collaborator) must be able to clear, with **zero communication cost**, to independently conclude that the project is engineered to a senior standard.

v2.5 enters the frozen baseline state when **every** item below is satisfied. Once frozen, v2.5 is not modified retroactively — any new work goes to v2.6.

---

## Acceptance Dimensions

Four orthogonal dimensions. A v2.5 release that fails any single dimension is not v2.5.

1. **Cold Clone Test** — a stranger can run it from a fresh clone
2. **Demo Readiness** — I can record a complete walkthrough end-to-end
3. **Documentation Completeness** — an interviewer can self-serve without asking me
4. **Stability Boundary** — v2.5 is a frozen baseline that v2.6 will not regress

---

## 1. Cold Clone Test

The hard gate. Any failure here means v2.5 is not done.

- [ ] **1.1 Environment prerequisites are explicit.** README's first screen lists minimum hardware (CPU / RAM / Disk), required software with versions (Docker, Docker Compose, Git), and OS compatibility matrix (macOS / Linux / WSL2). A stranger can decide within 30 seconds whether their machine qualifies.
- [ ] **1.2 One-command startup.** `git clone` → `cd` → one command (`make up` or `docker compose up -d`) brings up the full stack. No hidden steps such as "first export this env var" or "first create this directory."
- [ ] **1.3 First-run completes in under 15 minutes** (including image pulls). If this is unachievable, README explicitly states the expected duration and provides a "fast-validation subset" path.
- [ ] **1.4 Health-check surface exists.** After startup, a single URL (e.g. `localhost:xxxx/health` or a status dashboard) shows green / red status for every service (MinIO, Trino, Dagster, metastore, etc.). The stranger does not need to run `docker ps` to form judgment.
- [ ] **1.5 Hello-world data flow works out of the box.** A `make demo` (or equivalent) command runs one full ECB or DAX ingestion → Bronze → Silver → Gold → one Trino query, end-to-end. The stranger sees data flowing within 5 minutes of startup completion.
- [x] **1.6 Clean teardown.** A single command (`make down` or `docker compose down -v`) removes containers, volumes, and networks completely. No orphaned resources.

## 2. Demo Readiness

- [ ] **2.1 Single-take recording is possible.** A 20–30 minute walkthrough — cold start → three-layer data view → one compliance query → Dagster pipeline UI → lineage view — runs end-to-end without mid-recording manual fixes or window juggling.
- [ ] **2.2 `DEMO.md` exists with a fixed script.** Pacing is locked (e.g. "00:00–02:00 startup + architecture, 02:00–05:00 ingestion, ..."). Any future re-recording follows the same script.
- [ ] **2.3 Demo data is real, not mock.** Live ECB / DAX feeds drive the demo. This is the differentiator from tutorial projects and must not be compromised.
- [ ] **2.4 Failure fallbacks are documented.** `DEMO.md` flags at least one "if service X fails to start, skip this segment and pivot to Y" branch. Pretending nothing can fail is a junior signal.

---

## 3. Documentation Completeness

The interviewer reads the repo for 30 minutes and forms a judgment **without asking a single question**. Every doc must be self-explanatory.

- [ ] **3.1 README follows the three-part structure.** What (one sentence: SLH and the compliance pain it solves) → Why (why self-host, why compliance-first) → How (how to run). Avoid feature lists — interviewers want to see judgment, not inventory.
- [ ] **3.2 Architecture diagram is present and current.** A primary diagram (`docs/architecture/overview.png` or mermaid) plus `ARCHITECTURE.md` justifying component selection. Every component in the diagram has a corresponding explanation in the doc.
- [ ] **3.3 ADRs read as a coherent narrative.** All 14+ ADRs follow Context / Decision / Consequences / Alternatives Considered. Read in order, they form a continuous design-evolution story. **ADR-015 (MinIO → SeaweedFS deferral) explicitly explains the trade-off**, otherwise it reads as "abandoned mid-flight."
- [ ] **3.4 `RUNBOOK.md` exists.** Covers common operational scenarios: restarting a service, cleaning the metastore, inspecting Dagster logs, debugging a failed Trino query, handling a full disk. This is the strongest signal of "I can own this system in production."
- [ ] **3.5 Compliance narrative is a first-class artifact.** `docs/compliance/` contains three independent mappings — DORA, BaFin BAIT, MiFID II. Each explicitly states which SLH component / ADR maps to which regulatory clause. **This is the moat that distinguishes SLH from any other lakehouse demo project. It must be explicit, not implicit.**
- [x] **3.6 `CONTRIBUTING.md` and `LICENSE` exist.** Even with no contributors, their presence signals "real project, not a one-off demo."

## 4. Stability Boundary

The most important dimension: v2.5 must be a **frozen baseline that v2.6 will not need to revisit**.

- [ ] **4.1 CI is green.** GitHub Actions runs at least three classes of checks: (a) Docker Compose stack stands up, (b) `make demo` completes end-to-end, (c) key SQL queries return expected results. **If any of these are red, v2.5 is not done.**
- [x] **4.2 Tagged release with notes.** `v2.5.0` tagged on GitHub. Release notes list included features, known limitations, and explicit out-of-scope items (e.g. "MinIO → SeaweedFS migration: see ADR-015, planned for v2.7").
- [ ] **4.3 Issue list is clean.** Every v2.5-scope issue is closed or moved to the v2.6 milestone. **No open v2.5 issues remain** — an open issue means v2.5 is not actually done.
- [x] **4.4 Regression test set exists.** At least 5 critical-path scenarios (pytest or SQL assertions) automated. CI alarms if v2.6 work breaks any v2.5 core path. **This is the mechanical guarantee that v2.5 will not be regressed — it does not depend on discipline alone.**

## Meta-Acceptance: The Cold-Stranger Test

Before declaring v2.5 done, run **one external cold-start test**.

Hand the repo to a person who is *not* you — a Xiaohongshu technical follower, a non-Bernardo colleague, anyone with the relevant baseline. Give them 30 minutes, a coffee, and their own laptop. **You are not in the room.**

- If they bring it up and can articulate what SLH does → v2.5 passes.
- If they get stuck → wherever they get stuck is the real gap. Fix it before tagging.

## Out of Scope for v2.5

The following are explicitly **not** v2.5 deliverables. They live in v2.6 or later:

- MinIO → SeaweedFS migration (ADR-015 → planned post-DEP exam)
- ComplianceRAG module (v2.6 first AI module)
- LineageNarrator module (v2.7+)
- AuditCopilot module (v2.7+)
- Any new ingestion sources beyond ECB / DAX

Adding any of the above to v2.5 mid-flight is a scope-creep failure mode. Defer to v2.6.
